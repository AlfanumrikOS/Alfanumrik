**Architectural headline finding**: the app is mid-migration from a legacy Supabase-direct/RPC path to a server-driven `/v2` REST contract (generated OpenAPI/`dart-dio` client), gated end-to-end by the single compile-time flag `ApiConstants.useV2` (`--dart-define=USE_V2=true`, default **OFF**). When OFF, the app is described in-code as "byte-identical" to the pre-`/v2` behavior. Every repository, provider, and router redirect branches on this flag. This is the dominant structural fact for any future contract-validation work — two parallel code paths exist for quiz, learning, dashboard, and now routing/role-forking (parent tree).

## 3. Offline capabilities

| Capability | Mechanism | Files | Gating |
|---|---|---|---|
| Local response/data cache | Hive-backed `CacheManager`, 5-min TTL (`ApiConstants.cacheMaxAge`), JSON-serialized, per-key timestamp in a separate meta box | `core/cache/cache_manager.dart` | Always on (legacy path) |
| Auth/session cache | `current_student` Hive key stores only the `Student.toJson()` projection (explicitly NOT the raw `students` row, to avoid leaking phone/parent_phone/school_name/city/state — P13 note in-code) | `data/repositories/auth_repository.dart` | Always on |
| Offline quiz attempt capture + replay queue | `OfflineQuizStore` (Hive) queues a `QueuedQuizAttempt` (with immutable per-attempt `idempotencyKey` + `capturedAt`) when the device is offline at quiz-submit time; `OfflineDrainService` drains FIFO on reconnect via a `connectivityProvider` listener, classifying server responses into `success` / `discard` (permanent, e.g. `session_not_started`, `REPLAY_TOO_STALE`) / `retain` (5xx/network — retry, unchanged key) | `data/repositories/offline_quiz_store.dart`, `data/repositories/offline_drain_service.dart`, `providers/offline_quiz_provider.dart`, `data/models/offline_quiz_models.dart` | **`useV2` ON only** — offline replay does not exist on the legacy path |
| Offline quiz prefetch bundle | `OfflineQuizCoordinator.prefetchTodayBundle()` caches a server-shuffled session (questions, no correct answers — P6 preserved) per subject while online, for later offline attempt | `offline_quiz_provider.dart` | `useV2` ON only |
| Background sync trigger | **Connectivity-transition listener only** (`connectivityProvider` online-edge fires `coordinator.drain()`). No OS-level background task (no WorkManager/BGTaskScheduler equivalent) — drain only runs while the app is foregrounded or the Dart isolate is alive at reconnect | `providers/offline_quiz_provider.dart` | `useV2` ON only |
| Network reachability check | `connectivity_plus`-backed `hasConnection()` / `connectivityProvider` | `core/network/network_info.dart` | Always on |
| API retry (non-offline) | Dio interceptor: unconditional retry for pre-flight failures (timeout/connection error); 5xx retry restricted to idempotent methods (GET/HEAD/OPTIONS) only — mutations are never auto-retried to avoid double-writes | `core/network/api_client.dart` | Always on |

**Gap**: no true OS-level background sync (no `workmanager`/`android_alarm_manager_plus` dependency in `pubspec.yaml`). Offline-captured quizzes only drain when the app is running and connectivity flips, or on next app-foreground path that calls drain (not observed as an explicit app-lifecycle hook in the files reviewed — only the connectivity listener was found).

## 6. Auth flow, deep links, push/notifications, Play Store compliance

### Auth flow
- Supabase Auth, email/password, `AuthFlowType.pkce` (set in `main.dart`).
- No dedicated `/auth/callback` GoRouter route — relies on the native intent-filter deep link (below) + `supabase_flutter`'s own deep-link handling.
- 3-layer profile creation is NOT mirrored on mobile the way web's onboarding-integrity (P15) 3-layer failsafe is described — mobile signup does a single client-side `students` insert with no server-side bootstrap fallback observed in `auth_repository.dart`. **Potential P15 parity gap** — flagged for backend/architect follow-up, not fixed here.

### Deep links
- Android manifest intent-filter: `android:scheme="com.alfanumrik.app" android:host="callback"` — Supabase auth callback deep link, `autoVerify="true"`.
- Web→mobile route translation for the `/v2/today` server-emitted deep links: `today_deeplink.dart` (see §1).
- No iOS project found under `mobile/` in this pass (only `mobile/android/` was inventoried) — **iOS build/config was not discovered**; flagged as either not-yet-built or out of scope for this pass.

### Push / notifications
- **No push notification dependency found** (no `firebase_messaging`, no APNs/FCM wiring, no local-notification package) in `pubspec.yaml` or `mobile/lib/`. Web has a `/notifications` page (`src/app/notifications/page.tsx`) and a daily-cron notification producer; mobile has **no equivalent surface at all** — full parity gap.

### Play Store compliance artifacts
- `mobile/PLAY_STORE_LISTING.md` (140 lines): app name, descriptions, category, content-rating self-answers (target audience 13+, expected Everyone/PEGI3/IARC3+), contact email, visual-asset specs, and a **privacy-policy outline only** — the actual hosted Privacy Policy URL is a placeholder (`(required — see privacy_policy.md)`); no `privacy_policy.md` file was found via this pass's directory listing, so the required hosted URL may not exist yet.
- Listing copy mentions "Earn XP & Level Up" in the screenshot list (item 7) while the body copy above it (item "PERFORMANCE SCORE & FOXY COINS") already reflects the newer economy — **minor internal copy inconsistency** (stale XP framing vs the current Performance Score/Foxy Coins system), not a functional bug.
- `android/app/build.gradle`: `minSdkVersion 21`, `targetSdkVersion flutter.targetSdkVersion` (inherited from Flutter tooling, not pinned in this file), R8 minify + shrinkResources enabled for release, multidex enabled, ABI splits deferred to Flutter CLI flags (`--split-per-abi`/appbundle) rather than a manual `splits.abi` Gradle block (explicit in-code comment: AGP forbids both a manual splits.abi block and Flutter-injected `ndk.abiFilters` simultaneously — this was the exact regression REG-90 was created to catch, PR #957).
- Signing: `key.properties`/keystore conditionally loaded; falls back to debug signing when absent (CI/local), so unsigned CI release builds still succeed.
- `usesCleartextTraffic="false"` + `networkSecurityConfig` present (not read in this pass).

## 8. Ownership + gaps

### Web features with no mobile counterpart (parity gaps)
1. **Push notifications** — web has `/notifications` page + cron producer; mobile has zero notification infrastructure (no FCM/APNs dependency at all).
2. **Weekly Dive / Monthly Synthesis** (`/dive`, `/synthesis`) — explicitly not ported; `today_deeplink.dart` falls back these to `/learn`.
3. **True OS-level background sync** — offline quiz drain only fires on in-app connectivity-transition or app-foreground call sites, not a scheduled/OS background task.
4. **iOS build target** — not found under `mobile/` in this pass (Android-only artifacts discovered: `mobile/android/`).
5. **`DailyPlanRepository`** (`GET /api/student/daily-plan`) is implemented but has no discovered provider/screen wiring it up — looks like an unused/orphaned repository (possibly a landed-ahead-of-UI feature, Goal-Adaptive Daily Plan Phase 6 per its doc header).
6. **P15 onboarding 3-layer failsafe** — web's documented `client insert → /api/auth/bootstrap → AuthContext runtime fallback` 3-layer pattern does not appear mirrored in mobile's `auth_repository.dart` signup path (single client-side insert only). Flagged for backend/architect + mobile follow-up, not confirmed as a live bug in this read-only pass.

### Mobile-only behaviors
1. **STEM Lab WebView wrapper** (`stem_lab_screen.dart`) — authenticated WebView embed of web's `/stem-centre`, with Supabase session injected into WebView `localStorage` on every page load, and same-origin navigation lock (external links open in system browser). This is a mobile-specific integration pattern with no web equivalent to compare against.
2. **`useV2` compile-time flag architecture** — the dual-path (legacy Supabase-direct vs generated `/v2` client) pattern is unique to mobile's migration strategy; web has no equivalent single-flag fork of this shape.
3. **Offline quiz replay queue** (Hive-backed, idempotency-key-immutable) — no directly equivalent web offline-replay system was checked in this pass (web is browser-online-first).
4. **Role-aware router fork to a minimal parent tree** (`/parent`, glance-only, no bottom nav) — a mobile-specific "B2C-guardian-lite" experience distinct from web's fuller `/parent/*` (6 pages).

### Endpoints mobile calls that web has changed/removed
1. **`functions/v1/foxy-tutor` Edge Function** — mobile's `ChatRepository._sendViaEdge` legacy branch still targets this function via `Supabase.instance.client.functions.invoke('foxy-tutor', ...)`.

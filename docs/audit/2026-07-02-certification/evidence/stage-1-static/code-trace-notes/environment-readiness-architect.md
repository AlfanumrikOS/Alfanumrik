## VERDICT: STAGING IS ISOLATED FROM PRODUCTION — **PARTIALLY**

The **database layer** (Supabase project, credentials, migration/function sync paths) is isolated with HIGH confidence — different project, different org, different secrets, explicit fail-closed guards against the prod ref in every staging-write workflow. This part of the isolation story is solid.

The **hosting layer (Vercel)** and at least one **shared external dependency (Upstash Redis, and possibly Sentry/other Vercel-scoped env vars)** could **NOT be verified** from this repository + `gh` CLI alone — Vercel's per-environment (Production vs Preview) variable values live entirely in the Vercel dashboard, which this session has no credentialed access to. There is a concrete, non-hypothetical failure mode this gap could hide: a staging/Preview Vercel deployment whose `NEXT_PUBLIC_SUPABASE_URL` is still pointed at the **production** Supabase project (human misconfiguration, not a code defect) would make the DB-layer isolation moot for anything done through the deployed web app, even though the CI-driven migration/function/data-seed pipelines are correctly staging-scoped.

**Recommendation before any certification work touches staging**: obtain (from the user/ops, who has Vercel dashboard access) a confirmation of the actual `NEXT_PUBLIC_SUPABASE_URL` / `SUPABASE_SERVICE_ROLE_KEY` / `UPSTASH_REDIS_REST_URL` values configured on Vercel's **Preview** environment, and confirm they differ from the values on **Production**. This is the one fact this static/read-only pass cannot supply and that the whole isolation claim rests on for anything reached via the deployed staging URL (as opposed to direct `psql`/Supabase-CLI access, which IS independently verified safe below).

Additionally: no GitHub Environment in this repo has protection rules (re-confirmed independently below, matches ops' OPS-2 finding) — this doesn't break staging/prod *isolation* per se, but it does mean nothing in GitHub itself would stop a workflow that *did* target prod from running unattended. The safety in the staging-specific workflows comes entirely from the explicit in-script fail-closed ref checks (good, and independently verified below), not from any GitHub platform-level gate.

**Every one of these is a literally different secret name than its production counterpart** (`SUPABASE_ACCESS_TOKEN` vs `SUPABASE_STAGING_ACCESS_TOKEN`, `SUPABASE_PROJECT_REF` vs `SUPABASE_STAGING_PROJECT_REF`, etc.). Both `deploy-staging.yml` and `sync-staging-migrations.yml` carry inline comments stating the staging project "lives in a different Supabase org than prod" and that the prod token "returns Unauthorized" against it — this is asserted as an operational fact learned from a real failure, not speculation.

**Hard runtime fail-closed proof (not just naming convention)**: `staging-flag-set.yml:119-135` and `staging-adaptive-drill.yml:120-135` both hardcode the known PROD project ref (`shktyoxqhundlvkiwguu` — matches `supabase/config.toml:16`'s pinned `project_id`, confirmed by direct read) as `PROD_PROJECT_REF` and assert, **before any DB connection is opened**, that `SUPABASE_STAGING_PROJECT_REF != PROD_PROJECT_REF`. `staging-adaptive-drill.yml` goes further: after `supabase link`, it re-reads the CLI's own linked-ref state file (`supabase/.temp/project-ref`) and the ref *embedded in the CLI-derived pooler connection string itself* (`supabase/.temp/pooler-url`), and fails closed if either one matches the prod ref — this defends against the secret *value* silently being wrong even if the secret *name* is correct. This is a materially stronger guarantee than "the workflow file says staging" — it's a same-run, self-verifying assertion.

**Different Supabase org**: repeated inline comments across 4 files independently state staging is in a different Supabase org (not just a different project in the same org). I cannot independently verify org membership without Supabase dashboard/API access — this is corroborating documentation, not something I directly confirmed via `gh` — but it is consistent with the "prod token returns Unauthorized" operational evidence, which IS a hard technical signal (a same-org PAT would typically still authenticate, just fail authorization checks differently).

**Can I confirm the secret *values* differ?** No — `gh secret list` (and the environment-secrets API) only ever returns secret *names* and metadata (creation/update timestamps), never values, by GitHub design. I did NOT and cannot read secret values. What I *can* confirm, and did: the names are different, the workflows fail closed if the staging ref were ever accidentally equal to the known prod ref, and `.env.local`'s `NEXT_PUBLIC_SUPABASE_URL` (which the repo's own `supabase/config.toml` header comment labels "PROD + promotion target") resolves to project ref `shktyoxqhundlvkiwguu` — the exact same literal ref value that `staging-flag-set.yml`/`staging-adaptive-drill.yml` hardcode as the forbidden ref. This closes the loop: the "known prod ref" baked into the staging-safety workflows is demonstrably the *actual* prod ref this codebase's local dev environment points at, not a stale or wrong placeholder.

**Verdict for point 1: YES, isolated, HIGH confidence**, for the Supabase database/CLI/RPC access layer specifically.

## 2. Separate hosting target (Vercel) — Confidence: **LOW / NOT FULLY VERIFIABLE**

Read: `deploy-production.yml`, `deploy-staging.yml`, `vercel.json`, `docs/ops/deployment-flow.md`.

Findings, in order of certainty:

- **`vercel.json`** (root) contains no environment- or branch-specific configuration at all — it's a single shared file (region `bom1`, function timeouts, cron schedules) used by whatever Vercel project(s) build from this repo. It does not distinguish staging from production; it can't, by design (it's baked into the git tree, not per-deployment).

- **Both `deploy-production.yml` (workflow-level `env:`, lines 19-20) and `deploy-staging.yml` (workflow-level `env:`, lines 20-21) reference the SAME secret names**: `secrets.VERCEL_ORG_ID` and `secrets.VERCEL_PROJECT_ID`. This is the single piece of hard evidence on this point, and it points toward **staging and production sharing one Vercel project**, differentiated only by Vercel's own Production-vs-Preview environment concept within that project (`vercel deploy --prod` in `deploy-production.yml:247` vs plain `vercel deploy` / `vercel pull --environment=preview` in `deploy-staging.yml:121,133`).

The workflow's own comment (`deploy-production.yml:166-169`) states the canonical production web deploy is instead **Vercel's native GitHub App auto-deploy on push to `main`**, which is configured entirely inside the Vercel dashboard and is invisible to both this repo's YAML and to `gh api` (Vercel's GitHub App integration is a separate system from GitHub Environments/secrets). **This means the one piece of evidence I have (shared `VERCEL_PROJECT_ID` secret) does not actually prove what project the *currently-live* production deploy path targets** — it only proves what project the *dormant* CLI path would target if re-enabled. The staging CLI path (`deploy-staging.yml`), by contrast, IS the live path for staging and does use that shared secret.

- I found **no evidence of a second, staging-specific Vercel project** anywhere in the repo (no second `VERCEL_*_PROJECT_ID`-shaped secret, no `vercel-staging.json`, no per-environment Vercel config file). Absence of evidence is not proof of absence here, but it is consistent with (not contradictory to) the single-shared-project reading above.

- **Corroborating but stale**: `docs/ops/deployment-flow.md:146-150` (last verified 2026-04-02, i.e., 3 months stale relative to this audit) lists the environment-management table with Supabase Project as **"Same or separate project"** for Staging — the doc author was themselves uncertain on this exact question at the time, and it has not been updated since. This is weak corroboration that ambiguity has existed here before, not an authoritative answer.

- **Env-var scoping evidence found, but the values are exactly what's unverifiable**: the same doc (`deployment-flow.md:122-137`) states Vercel env vars including `NEXT_PUBLIC_SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY`, `UPSTASH_REDIS_REST_URL`/`_TOKEN`, `NEXT_PUBLIC_SENTRY_DSN` are "Configured per environment (Production / Preview / Development)" in the Vercel dashboard. `deploy-staging.yml:18-19` independently confirms Vercel's per-environment env var pull is real and load-bearing: the workflow-level `env:` block hardcodes an obvious placeholder `NEXT_PUBLIC_SUPABASE_URL: 'https://placeholder.supabase.co'` purely so the `quality` job's `next build` step can compile without real credentials — the *actual deployed* Preview build instead runs `vercel pull --yes --environment=preview` (`deploy-staging.yml:120-121`) to fetch whatever is really configured for Preview in the Vercel dashboard, then builds/deploys with that. **I cannot see what that Preview-scoped `NEXT_PUBLIC_SUPABASE_URL` (or `SUPABASE_SERVICE_ROLE_KEY`, or `UPSTASH_REDIS_REST_URL`) actually resolves to** — that's set in a UI this session has no token for. If it were still pointed at the production Supabase project (a plausible one-time misconfiguration, not something the code would catch — `deploy-production.yml:205-233` DOES have a guard that fails the *production* build if it detects the *placeholder* string, but there is no equivalent guard anywhere that fails a *Preview* build for pointing at *prod's real* URL, since a real prod URL is not distinguishable from a real staging URL by string-pattern alone), then every certification action performed through the deployed staging web app (not the direct Supabase CLI/psql channel) would silently write to production regardless of how careful the CI-side migration/secret separation is.

**Verdict for point 2: NOT VERIFIABLE with the access this session has.** The one concrete signal (shared `VERCEL_PROJECT_ID` secret) leans toward "same Vercel project, Production vs Preview environment" rather than "two separate Vercel projects" — which is a normal, common, and often-fine Vercel setup, but it means the actual isolation boundary for anything reached through the *deployed web app* is Vercel's dashboard-side per-environment variable scoping, not a project boundary — and that dashboard-side configuration is exactly the piece this static/read-only pass cannot see. **This is the single highest-priority open item before certification touches anything through the staging web app URL** (as opposed to direct Supabase CLI/psql access via the `SUPABASE_STAGING_*` credentials, which is independently verified safe above and below).

## 3. No shared resources — Confidence: **MIXED (Redis: LOW/NOT VERIFIABLE; Sentry: MEDIUM; cron: HIGH-not-shared; second deploy pipeline: HIGH-found-but-orthogonal)**

- **Redis (Upstash)**: Grepped all of `.github/workflows/*.yml` for `UPSTASH` — **zero matches**. Redis configuration is never referenced by any GitHub Actions workflow (staging or production); it must be Vercel-dashboard-configured per environment, same as the Supabase URL above. `.env.local` (which, per `supabase/config.toml`'s own header comment, points at the PROD Supabase project) contains a single `UPSTASH_REDIS_REST_URL`/`UPSTASH_REDIS_REST_TOKEN` pair. I am not disclosing the value here (per instructions not to leak secret values into this report), but its mere presence in the prod-pointed `.env.local` combined with zero GitHub Actions references means **I cannot confirm from this repo whether the Vercel Preview environment has a separate Upstash Redis instance configured, or reuses the same one** `.env.local`/production uses. Rate-limiting behavior (`src/proxy.ts` per `CLAUDE.md`) falls back to in-memory if Redis is unreachable, so a shared Redis would not be a correctness bug, but it WOULD mean rate-limit counters (and anything else stored in that Redis) are not isolated between environments during certification load-testing. **NOT VERIFIABLE from this session; flag as an open item for whoever has Vercel dashboard access.**

- **Sentry**: `sentry.client.config.ts`, `sentry.server.config.ts`, `sentry.edge.config.ts` (confirmed via ops' full read, corroborated by my own grep of `next.config.js:25,243-244`) initialize `dsn: process.env.NEXT_PUBLIC_SENTRY_DSN` and set `environment: process.env.NODE_ENV || 'development'` (`sentry.server.config.ts:14`, directly grepped by me). Two relevant facts: (a) whether Preview and Production point at the *same* Sentry DSN/project is, again, a Vercel-dashboard env-var question I cannot see; (b) even if they share one DSN, every Sentry init tags events with `environment` — and Vercel sets `NODE_ENV=production` for BOTH Production and Preview builds by default (this is standard Next.js/Vercel behavior, not something this repo's code overrides for `NODE_ENV` specifically — `next.config.js:68` does separately expose `NEXT_PUBLIC_VERCEL_ENV` from Vercel's own `VERCEL_ENV`, which correctly distinguishes `production`/`preview`/`development`, but I did not find that finer-grained var actually wired into the Sentry `environment:` tag in the three config files — they use `NODE_ENV`, not `NEXT_PUBLIC_VERCEL_ENV`). **Practical implication**: if Preview and Production share one Sentry DSN, Preview-environment errors during certification testing would likely show up tagged `environment: production` in Sentry, not distinguishably as staging. Confidence: MEDIUM (the DSN-sharing question is unverified; the `NODE_ENV`-not-`VERCEL_ENV` tagging fact is directly confirmed by grep).

- **Cron schedules**: `vercel.json`'s 13 cron entries (`crons:` array, lines 32-84) are Vercel-project-scoped — Vercel only executes `crons` for the project's **Production** environment/deployment, never for Preview deployments (this is documented, stable Vercel platform behavior, not something visible in this repo's code — flagging as a platform-behavior fact I'm relying on rather than something I independently verified via `gh`/file read). Given that, **if staging/Preview is a separate Vercel *environment* within the same project (per point 2's leading hypothesis), staging would not run these crons at all** — so no cron-driven cross-environment write risk exists on that reading. If staging were somehow its OWN separate Vercel *project* with its own `vercel.json` copy and its own cron config, it would need its own separate cron schedule, and I found no evidence of a second `vercel.json` anywhere in the repo. Net: **no evidence of shared/colliding cron execution between staging and production** either way. Confidence: HIGH that crons don't cross-fire, contingent on standard Vercel behavior.

- **Second deployment pipeline (found while investigating, orthogonal to staging but relevant to the overall isolation picture)**: `deploy-aws.yml` exists and, per ops' independently-confirmed finding (`ops-findings.md` OPS-3, which I did not re-verify myself this pass but which cites specific `gh variable list` output: `ENABLE_AWS_DEPLOY=true`, `PRODUCTION_DOMAIN=https://da8yhieheuw7p.cloudfront.net`), is **currently live** — building a Docker image and deploying to AWS ECS/CloudFront on every push to `main`, in parallel with the Vercel production deploy. I did not find any staging-equivalent AWS workflow (no `deploy-aws-staging.yml` or similar in the workflow listing). This means the AWS pipeline is prod-only and does not itself create a staging/prod isolation gap — but it is a second live production surface the certification exercise should be aware of when defining "production," since `alfanumrik.com` (Vercel-served, per ops' independent DNS/curl check) and the CloudFront domain are two different currently-deployed targets of the same `main` branch.

**Verdict for point 3: PARTIALLY confirmed / PARTIALLY not verifiable.** No shared-resource risk was *found*, but for Redis and Sentry specifically, "not found via GH Actions" only proves the CI layer doesn't reference them — it says nothing about Vercel-dashboard-level sharing, which is the same blind spot as point 2.

Ran (read-only): `gh api repos/AlfanumrikOS/Alfanumrik/environments`, `gh api .../environments/production` (lowercase), `gh api .../environments/Production/secrets`, `gh api .../environments/staging/secrets`, `gh api .../environments/Preview/secrets`, `gh api .../environments/{Production – alfanumrik, Production – alfanumrik-learning-os}/secrets`, `gh api .../deployments?per_page=10`.

- **Case-normalization confirmed**: `gh api repos/AlfanumrikOS/Alfanumrik/environments/production` (lowercase, matching the exact string used in `deploy-production.yml:149,174,390`) returned the identical environment object (same `id: 12976220548`, `name: "Production"`) as querying `.../environments/Production` directly. (I did not test whether protection-rule *enforcement* itself is case-sensitive vs. just secret resolution — moot here since there are no protection rules configured on either casing to test.)

- **`staging` environment has zero environment-scoped secrets** (`{"total_count":0,"secrets":[]}`, confirmed directly). The actual `SUPABASE_STAGING_*` / `STAGING_SUPABASE_*` secrets referenced by staging workflows are **repo-level** secrets (visible in the flat `gh secret list` output), not scoped to the `staging` GitHub Environment. Repo-level and environment-level secrets are additive in GitHub's model, so this doesn't break anything functionally — the workflows still resolve the secrets correctly — but it does mean declaring `environment: staging` in these workflows currently buys **zero** additional secret-scoping protection beyond what any other workflow in the repo already has access to by default. This is a defense-in-depth gap (repo-level secrets are reachable by any workflow run, not just ones that declare `environment: staging`), not an isolation gap between staging and prod specifically — the staging secrets are still a completely different set of names/values than the prod-scoped ones regardless.

**Verdict for point 6: YES, CERT-03 re-confirmed still true, HIGH confidence.** No environment has protection rules; deploys and any staging-write workflow run fully automatically/unattended once triggered.

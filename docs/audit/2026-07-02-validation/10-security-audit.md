**Verdict: REFUTED (for the IDOR/data-exposure concern) â€” CONFIRMED (for the narrower architectural-inconsistency framing only)**

**Callers (confirmed):**

- Both functions are called directly from client React pages via the Supabase JS SDK, never proxied through a Next.js `route.ts` â€” confirms discovery Â§5.6's observation that this bypasses the `authorizeRequest`/permission-code layer entirely.

**Internal auth each Edge Function verifies (read in full):**
- `supabase/functions/parent-portal/index.ts:1360-1419` (`Deno.serve` dispatcher) â€” requires `Authorization: Bearer <token>` on every request (401 if missing), calls `supabase.auth.getUser(token)` (401 on invalid/expired), then for every action except `parent_login` resolves the canonical `guardian_id` from `guardians.auth_user_id = <JWT user.id>` and **overwrites `body.guardian_id`** with the trusted value (line 1419: `body.guardian_id = guardian.id`) â€” a request-supplied `guardian_id` can never be used to impersonate another guardian. `parent_login` itself takes the JWT-derived `authUserId` (not `body.auth_user_id`) per the explicit P13 fix documented at lines 1379-1384 ("body.auth_user_id is now ignored" â€” this is the historical PR #591 hardening referenced throughout the file).
- Every per-child data action (`get_child_dashboard` line 887-904, `get_monthly_report` line 1159-1176, `get_child_attendance` line 1096-1107) explicitly re-verifies a `guardian_student_links` row with `status IN ('active','approved')` for the exact `(guardian_id, student_id)` pair **before** any student data query runs, returning HTTP 403 `"You do not have access to this child's data."` on failure. `get_children`/`get_all_children_dashboard` scope by the JWT-resolved `guardian_id` only (never take a student_id from the request), so there's no cross-guardian read surface there either.
- `supabase/functions/teacher-dashboard/index.ts:3705-3758` â€” identical pattern: `resolveTeacherFromJwt()` requires a Bearer token, resolves `teachers.id` from `auth_user_id = user.id`, 403s ("Caller is not a teacher") if none, and the dispatcher **overwrites `body.teacher_id`** with the JWT-derived value before any handler runs (line 3758). - Tenant scoping (TSB-1): the file's own grade-fallback paths (`resolveTeacherSchoolId()` line 85-101) additionally scope any grade-based student query to the teacher's own `school_id`, fail-closed (empty result) for school-less/independent teachers â€” preventing a cross-school leak via the grade pseudo-class path.

**IDOR test (can a student invoke this for another student's data?):** No. A caller must hold a valid Supabase session JWT. For `parent-portal`, non-`parent_login` actions require an existing `guardians` row matching that JWT's `auth_user_id` (a student's own JWT has no matching guardian row â†’ 403 "Caller is not a registered guardian"). Even a caller who *does* have a guardian row can only reach a child's data through an `active`/`approved` `guardian_student_links` row â€” and new links created via `parent_login` are inserted as `status: 'pending'`, requiring the **student's own explicit approval** (Option B consent gate, `notifyStudentOfPendingLink` + `/api/parent/approve-link`) before any data becomes reachable; brute-forcing another student's link code is additionally rate-limited to 5/hour/IP (`PARENT_LOGIN_IP_LIMIT`, line 87-89). For `teacher-dashboard`, a caller must hold a `teachers` row matching their JWT, and every handler that takes a `class_id`/`student_id`/`poll_id`/`alert_id` re-verifies ownership before querying â€” confirmed by direct code read of `assertTeacherOwnsClass`, `assertTeacherOwnsPoll`, and `resolveStudentsForTeacher`/`teacherOwnsAssignment` gates on every handler in the file.

**Severity: S3 (low)** â€” no exploitable IDOR found; the "bypasses Next.js RBAC" framing is architecturally true (no `authorizeRequest('permission.code')` call â€” these functions implement a bespoke JWT-binding + per-resource-ownership scheme instead) but functionally equivalent in security outcome, and in places (JWT-derived identity override of body-supplied ids, fail-closed tenant scoping) demonstrably more defensive than a naive `authorizeRequest` call would be on its own. This is a maintainability/consistency finding, not a vulnerability: a future handler added to either file without copying the existing ownership-check pattern would silently reintroduce IDOR, since there's no framework-level enforcement (unlike `authorizeRequest` routes, which fail closed by construction).

**Exploit scenario:** None currently reachable. Theoretical: a future PR adds a new `case` to either dispatcher's `switch` that takes a `student_id`/`class_id` from `body` and queries with the service-role client without calling the existing `assertTeacherOwnsClass`/link-check helpers â€” this would reintroduce cross-tenant IDOR silently, since there is no compiler/framework/lint gate forcing the ownership check (unlike Next.js RBAC routes where omitting `authorizeRequest` is at least a discoverable pattern deviation reviewers are primed to look for, per this codebase's own review-chain conventions).

**Remediation (do not implement):** Not a bug to fix now. Recommend a lint rule or code-review checklist item flagging any new `case` added to `parent-portal`/`teacher-dashboard`'s dispatcher that reads a body-supplied `*_id` without a matching ownership-check call in the same handler; longer-term, converge these two Edge Functions onto the same `canAccessStudent`-style shared authz primitive referenced in the file's own TSB-3 TODO (teacher-dashboard:3840) once a Next.js/Deno bridge exists.

## G-3: src/app/api/oauth/authorize real auth posture â€” STATUS: done

**Verdict: REFUTED â€” no user-session check is present, but this is safe because the route never issues an authorization code at all; the codebase's `authorization_code` grant is an explicit unimplemented stub.**

**Evidence:**

- `src/app/api/oauth/token/route.ts:118-167` (`handleAuthorizationCodeGrant`) â€” after validating `client_id`/`client_secret` (constant-time `secureEqual()`, line 107), the function's own comment block (lines 139-166) states verbatim: *"Authorization code exchange placeholder... **Exploit scenario:** None reachable today. If someone tried to exploit "no auth check on `/oauth/authorize`," the furthest they could get is reading back an approved app's public name/logo/description and a school's name/logo (all discoverable via the same consent screen a legitimate user would see) â€” no code, token, or student data is ever exposed by this endpoint. **Remediation (do not implement):**
1. For the authorize endpoint itself: no fix needed â€” behavior matches OAuth2 spec intent (unauthenticated consent-screen data lookup); optionally rename/comment the route to make explicit "this endpoint never issues a code" so a future contributor doesn't assume it does.

## G-6: payments/setup-plans inline x-admin-secret compare â€” STATUS: done

**Verdict: CONFIRMED (the deviation from `requireAdminSecret()` is real) but REFUTED as a vulnerability â€” the compare is timing-safe and the route is gated by a strictly more powerful secret than the standard admin-secret helper, not a weaker one.**

**Evidence:**
- `src/app/api/payments/setup-plans/route.ts:17-27` â€” reads `x-admin-secret` from the request header and compares it against **`process.env.SUPABASE_SERVICE_ROLE_KEY`** (not `SUPER_ADMIN_SECRET`), using `secureEqual()` (line 25: `if (!adminSecret || !serviceKey || !secureEqual(adminSecret, serviceKey))`) â€” **the same constant-time helper** used by the shared `requireAdminSecret()` in `src/lib/admin-auth.ts:404-416`. In-file comment (lines 21-24) explicitly documents this as intentional: *"Simple admin auth â€” only callable with the service role key as header... Constant-time compare... same fix as PR #610 admin-secret + cron-secret gates."*
- **Timing-safe: YES.** `secureEqual()` is used identically to the shared helper; this is not a naive `!==` compare.
- **Reachable in prod: YES.** No feature flag, no environment gate beyond the two required env vars (`SUPABASE_SERVICE_ROLE_KEY`, `NEXT_PUBLIC_SUPABASE_URL`) that are already mandatory in production per `CLAUDE.md`'s required-env-var list. The route compiles and is routable at `POST /api/payments/setup-plans` in every environment.
- **Deviation from the shared helper is real but the substituted secret is *more* privileged, not less.** `requireAdminSecret()` gates against `SUPER_ADMIN_SECRET` (a dedicated admin-auth secret used across 11 `internal/admin/*` routes per discovery Â§2.12). This route instead gates against the raw `SUPABASE_SERVICE_ROLE_KEY` â€” the single most powerful credential in the system (bypasses RLS entirely, full read/write on every table via direct PostgREST calls). Anyone who already possesses that key does not need this route to escalate privilege: they already have unrestricted DB access via `Authorization: Bearer <service_role_key>` against PostgREST directly, which is strictly more powerful than anything `setup-plans` can do. So while using two different secret systems across the admin-secret surface is a real **consistency** defect (a future engineer copy-pasting this file's pattern for a new route, believing it's the standard `x-admin-secret` convention, would accidentally gate a route behind the service-role key instead of `SUPER_ADMIN_SECRET`), it does not itself create a privilege-escalation path â€” someone who has the service-role key already has god-mode.
- **What it can mutate:** for every `is_active=true, price_monthly>0` row in `subscription_plans`, calls `createRazorpayPlan()` (external Razorpay API call, `src/lib/razorpay.ts`) to create a monthly plan (if `razorpay_plan_id_monthly` is NULL) and a quarterly plan (if `razorpay_plan_id_quarterly` is NULL), then writes the returned Razorpay plan id back onto the row (lines 66-69, 90-93). It **only fills NULL columns â€” never overwrites an existing plan id** ("Safe to call multiple times â€” skips plans that already have IDs," confirmed by the `already_exists`/`created` status branching at lines 56-97). Blast radius: platform-wide pricing/billing-plan catalog (affects every future subscriber checking out on an affected plan), not per-student â€” but self-limiting because it cannot clobber an already-provisioned plan id, only backfill missing ones.
- **Tested:** `src/__tests__/api/payments/status-and-setup-plans.test.ts:14 ("NOT open â€” it requires the x-admin-secret header to equal the service..."), 185-192` explicitly assert 401 on missing/wrong `x-admin-secret` with no Razorpay plan created â€” this behavior is deliberate and covered, not an oversight.

**Exploit scenario:** Requires possession of the production `SUPABASE_SERVICE_ROLE_KEY` (env-var-only, never `NEXT_PUBLIC_`, never sent to any client per P8). Anyone who has that key can already read/write every table in the database directly â€” using it to call `setup-plans` instead would only let them backfill missing Razorpay plan ids on active paid plans, which is a strictly smaller capability than what the key already grants them. Not independently exploitable; the real risk is upstream (service-role-key leakage), same as every other route protected by that key.

**Severity: S3 (low)** â€” no privilege escalation possible; the finding is a code-consistency/maintainability issue (secret-system fragmentation), not a security hole. Downgraded from the task's implied "vs the shared helper" framing because the substituted secret is a superset of `SUPER_ADMIN_SECRET`'s privilege, not a bypass of it.

**Remediation (do not implement):** Migrate this route onto the shared `requireAdminSecret()` helper (i.e., gate on `SUPER_ADMIN_SECRET` like the other 11 `x-admin-secret` routes) so the platform has exactly one admin-secret convention; this also lets ops rotate `SUPER_ADMIN_SECRET` for routine plan-setup operations without needing the service-role key on hand, narrowing who needs to hold the most powerful credential day-to-day.

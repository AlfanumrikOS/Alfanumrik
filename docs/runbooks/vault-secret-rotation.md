# Runbook — Vault secret rotation

**Scope:** Supabase Vault secrets that pg_cron uses to invoke Edge Functions. NOT Vercel environment variables, NOT Supabase function secrets set via the dashboard. See note in §6 for the broader secret picture.

**Triggered by:**
- Scheduled rotation cadence (every 90 days; tracked in operations calendar)
- Suspected compromise of the service-role JWT
- A Supabase project-key rotation (Supabase admin rotates project keys; all derived JWTs must be regenerated)
- Edge Function URL changes (rare — happens if function is renamed or moved to a different project)

**Owner:** ops + architect.

## 1. What secrets are managed in Vault

Vault is the encrypted secret store inside Supabase Postgres. Used here because pg_cron jobs cannot reach Vercel env vars and the managed `postgres` role cannot use `ALTER DATABASE … SET app.*` (SQLSTATE 42501 — see comment block in [`20260524110002_projector_runner_cron.sql`](../../supabase/migrations/20260524110002_projector_runner_cron.sql)).

Current Vault secrets:

| Name | Purpose | Created by |
|---|---|---|
| `projector_runner_url` | URL of `projector-runner` Edge Function called by pg_cron every 1 min | operator, post [PR #753](https://github.com/AlfanumrikOS/Alfanumrik/pull/753) |
| `projector_runner_service_role_key` | Service-role JWT used as `Authorization: Bearer …` header by both projector cron jobs | operator, post [PR #753](https://github.com/AlfanumrikOS/Alfanumrik/pull/753) |
| `projector_health_check_url` | URL of `projector-health-check` Edge Function called by pg_cron every 2 min | operator, post Phase-5 Iter. 2 (this codebase) |

Future Vault secrets follow the same pattern: per-function URL secret + (optional) reuse of the shared service-role JWT secret.

## 2. Read the current state

Run from Supabase SQL editor against the target environment:

```sql
SELECT name, created_at, updated_at
FROM vault.decrypted_secrets
WHERE name LIKE 'projector_%'
ORDER BY name;
```

Expected: three rows. Missing rows mean the cron jobs that use them are silently failing — see [`projector-failure.md`](./projector-failure.md) step 4.

To inspect the actual secret value (for verification before rotation):

```sql
SELECT name, decrypted_secret
FROM vault.decrypted_secrets
WHERE name = '<secret name>'
LIMIT 1;
```

The decrypted value is only visible to roles with explicit grant on `vault.decrypted_secrets`. Service role has it; the dashboard SQL editor runs as service role when "Run with: postgres" is selected. Do **not** paste the decrypted value into chat / Slack / a PR.

## 3. Rotate a URL secret

URL changes happen if the Edge Function is renamed or the project is moved. The rotation is a single delete + recreate:

```sql
-- 1. Take a note of the old value somewhere local; for forensic reference.
SELECT decrypted_secret FROM vault.decrypted_secrets WHERE name = 'projector_runner_url';

-- 2. Delete the old secret. Vault's CRUD is via the helper functions —
--    do NOT manually DELETE FROM vault.secrets; that table has internal
--    columns Vault manages.
SELECT vault.update_secret(
  (SELECT id FROM vault.secrets WHERE name = 'projector_runner_url'),
  'https://<new-project-ref>.supabase.co/functions/v1/projector-runner',
  'projector_runner_url',
  'URL of the projector-runner Edge Function called by pg_cron'
);

-- 3. Verify the new value.
SELECT name, decrypted_secret FROM vault.decrypted_secrets
WHERE name = 'projector_runner_url';
```

The pg_cron job picks up the new URL on its next tick — no migration re-apply needed. The cron command reads the secret at execution time, not at schedule time.

## 4. Rotate the service-role JWT

This is the higher-risk rotation. Every consumer of the JWT must be updated atomically, or some cron will fail until they all converge.

**Pre-flight: identify all consumers.**

```sql
-- List all cron jobs to confirm which use the service-role JWT.
SELECT jobid, jobname, schedule, command
FROM cron.job
WHERE command LIKE '%projector_runner_service_role_key%';
```

Expected: 2 rows (`projector-runner-tick`, `projector-health-check-tick`) if Iteration 2 has shipped; 1 row otherwise. Add to this list as more cron jobs ship.

**The rotation:**

1. **Generate the new JWT.** Supabase dashboard → Project Settings → API → "Service role secret" → "Reveal" → copy. The new key is generated by Supabase; we don't construct JWTs ourselves.

2. **Update the Vault secret to the new value.**

   ```sql
   SELECT vault.update_secret(
     (SELECT id FROM vault.secrets WHERE name = 'projector_runner_service_role_key'),
     '<new service-role JWT — never paste in PR or chat>',
     'projector_runner_service_role_key',
     'Bearer token (service-role JWT) for projector-runner Edge Function. Rotated <YYYY-MM-DD> by <operator>.'
   );
   ```

3. **Verify the change took effect.** Force the next cron tick by invoking `tickOne` manually, or wait one minute and check:

   ```sql
   SELECT job_pid, status, return_message, start_time, end_time
   FROM cron.job_run_details
   WHERE jobname IN ('projector-runner-tick', 'projector-health-check-tick')
   ORDER BY start_time DESC
   LIMIT 5;
   ```

   Expected: `status = 'succeeded'` for ticks after the update. If `status = 'failed'` with `HTTP 401` in `return_message`, the new JWT is malformed or wasn't accepted — restore from step 5.

4. **Update any out-of-band consumers.** Any operator scripts, monitoring tools, or local development environments that use the old JWT must be updated. Search local notes; if in doubt, search runbooks for `service_role_key`:

   ```bash
   git grep -i 'service_role_key' docs/
   ```

5. **Rollback (if needed):** if step 3 fails, recover by re-running step 2 with the previous JWT value (the one you noted from step 2 of §2). The cron's next tick picks up the restored value.

## 5. Add a new Vault secret (new Edge Function)

When a new Edge Function lands that needs a cron schedule, follow the projector-health-check pattern:

```sql
SELECT vault.create_secret(
  'https://<project-ref>.supabase.co/functions/v1/<new-function-name>',
  '<new_function_name>_url',
  'URL of the <new-function-name> Edge Function called by pg_cron'
);
```

If the function calls require service-role auth (which they always do for cron-triggered functions), reuse `projector_runner_service_role_key` rather than creating a new JWT secret — the JWT is project-scoped, so one secret serves all functions.

Then create the cron migration following the [`20260526100000_projector_health_check_cron.sql`](../../supabase/migrations/20260526100000_projector_health_check_cron.sql) template — it has the pg_cron extension guard, the URL-availability guard, and the idempotent `cron.unschedule` + `cron.schedule` pattern.

## 6. Other secrets (out of scope for this runbook)

Not covered here:

- **Vercel environment variables** (`SUPABASE_SERVICE_ROLE_KEY` for Next.js BFF, `RAZORPAY_*`, `POSTHOG_PROJECT_API_KEY`, `VERCEL_API_TOKEN`, `SUPER_ADMIN_SECRET`) — rotated via Vercel dashboard. Rotation procedure depends on the secret's blast radius; treat each as its own runbook.
- **Supabase function secrets** (email provider credentials for `send-auth-email`, etc.) — set via `supabase secrets set` per [`CLAUDE.md`](../../CLAUDE.md). Rotate via the same command.
- **Razorpay webhook secret** (`RAZORPAY_WEBHOOK_SECRET`) — rotated jointly with Razorpay dashboard. See [`payment-webhook-recovery.md`](./payment-webhook-recovery.md).
- **Supabase Auth JWT secret** — rotated by Supabase (managed). Triggering this invalidates every active session; coordinated rotation only.

## 7. Verification checklist after any rotation

- [ ] `vault.decrypted_secrets` shows the updated row with `updated_at` matching now
- [ ] `cron.job_run_details` shows `status = 'succeeded'` for at least 2 consecutive ticks after the rotation
- [ ] No `[projector-runner] fatal:` or `[projector-health-check] fatal:` lines in Edge Function logs after the rotation
- [ ] `public.subscriber_lag` query (per [`projector-failure.md`](./projector-failure.md) §1) shows no spike in `age_behind` straddling the rotation timestamp
- [ ] `audit_logs` has a row recording the operator action:

  ```sql
  INSERT INTO public.audit_logs (actor_auth_user_id, action, target_table, target_id, metadata)
  VALUES (
    '<operator auth uuid>',
    'ops.vault_secret_rotated',
    'vault.secrets',
    '<secret name>',
    jsonb_build_object(
      'reason',  '<scheduled | suspected_compromise | project_key_rotation | function_renamed>',
      'runbook', 'docs/runbooks/vault-secret-rotation.md',
      'verified_at', NOW()
    )
  );
  ```

## 8. Rotation cadence

| Secret class | Cadence | Notes |
|---|---|---|
| Service-role JWT | 90 days OR on suspected compromise | Higher risk → tighter cadence on suspected compromise |
| URL secrets | On function rename / project move only | URLs do not "expire"; no scheduled rotation |
| Razorpay webhook secret | Annually OR on Razorpay-recommended cadence | Coordinate with Razorpay; out of scope for this runbook |

Calendar entry for next service-role JWT rotation lives in the operations calendar; runbook references that calendar, not a hardcoded date here.

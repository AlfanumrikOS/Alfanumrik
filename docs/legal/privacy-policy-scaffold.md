## 11. Security

We implement reasonable security practices and procedures appropriate to the nature of the personal data we hold, in compliance with Section 8 of the DPDP Act and the SPDI Rules.

Our safeguards include:
- **Encryption in transit** (TLS 1.2+) for all client–server traffic;
- **Encryption at rest** for the database and file storage (AES-256);
- **Row-Level Security (RLS)** enforced in the database for all student, parent, and teacher data, so that users can access only their own (or their authorized scope's) records;
- **Role-Based Access Control (RBAC)** with [FOUNDER FILL: e.g., "6 roles and 71 granular permissions"] enforced on every API endpoint;
- **Service-role isolation** — privileged keys are never shipped to client devices;
- **PII redaction in logs** — passwords, tokens, emails, phone numbers, and API keys are stripped from server logs and Sentry events before storage;
- **Webhook signature verification** for all payment events;
- **Anti-cheat and rate-limiting** to prevent abuse;
- **Regular backups** with periodic restore drills;
- **Vulnerability monitoring** and timely patching of dependencies;
- **Internal access** to production data restricted to authorized engineers under the principle of least privilege, with audit logging.

Despite these measures, no system can be guaranteed 100% secure. In the event of a personal data breach, we will notify the Data Protection Board of India and affected users in accordance with the DPDP Act and any rules thereunder.

### 11.1 IP-Address Handling (Carve-Out)
IP addresses are **redacted** from all telemetry destinations (Sentry, PostHog, the Anthropic Claude API, and similar third-party analytics or monitoring services) before any event leaves our infrastructure. However, IP addresses **are retained** in our internal security audit logs (the `audit_*` family of database tables, accessible only via service-role credentials and to administrators with the relevant RBAC permission) for the limited purpose of incident investigation, fraud detection, and abuse response. Audit-log retention follows the schedule set out in Section 8.

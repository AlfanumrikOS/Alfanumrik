# Route Access Contract

**Status**: Behavioral specification
**Date**: 2026-04-02
**Layer**: Delivery (Layer 4 of the identity mechanism)
**Depends on**: Auth (Layer 1), App Identity (Layer 2), Access Control (Layer 3)

---

## 1. Route Classifications

Every route in the application falls into exactly one of five categories. There is no ambiguity. If a route does not appear in this document, it is treated as **Protected + Onboarding Required** by default.

| Category | Auth Required | Onboarding Required | RBAC Required | Count |
|----------|:---:|:---:|:---:|:---:|
| Public | No | No | No | ~15 |
| Auth Flow | No (or partial) | No | No | ~8 |
| Protected | Yes | Yes | No | ~20 |
| Role-Restricted | Yes | Yes | Yes (specific role) | ~25 |
| Admin | Yes | N/A (admins skip onboarding) | Yes (admin/super_admin) | ~22 |

## 2. Public Routes

No authentication required. Accessible to everyone, including bots and crawlers.

| Route | Purpose | Notes |
|-------|---------|-------|
| `/welcome` | Landing page | Marketing content, sign-up CTA |
| `/login` | Sign-in form | Redirects to dashboard if already authenticated |
| `/signup` | Sign-up form | Redirects to dashboard if already authenticated |
| `/about` | About page | Static |
| `/pricing` | Pricing page | Static content, may include dynamic plan data |
| `/contact` | Contact form | Static with form submission |
| `/privacy` | Privacy policy | Static |
| `/terms` | Terms of service | Static |
| `/sitemap.xml` | SEO sitemap | Generated by `src/app/sitemap.ts` |
| `/manifest.json` | PWA manifest | Static |
| `/sw.js` | Service worker | Static |
| `/robots.txt` | Crawler rules | Static |
| `/api/v1/health` | Health check endpoint | Returns system status, no auth |
| `/api/auth/callback` | Auth callback | Handles PKCE exchange and token hash (see below) |

**Middleware behavior for public routes**: Session refresh is attempted (to keep authenticated users logged in) but not required. Security headers are applied. Rate limiting is applied. No auth check.

## 3. Auth Flow Routes

These routes are part of the authentication process. They have special handling because the user may be partially authenticated.

| Route | Auth State | Purpose | Redirect If Authenticated |
|-------|------------|---------|---------------------------|
| `/login` | Unauthenticated | Sign in | Yes, to role dashboard |
| `/signup` | Unauthenticated | Create account | Yes, to role dashboard |
| `/auth/callback` | Transitioning | PKCE code exchange, token hash | Proceeds to bootstrap/redirect |
| `/auth/confirm` | Unauthenticated | Email confirmation landing | Proceeds to callback |
| `/forgot-password` | Unauthenticated | Request password reset | Yes, to role dashboard |
| `/reset-password` | Authenticated (via recovery token) | Set new password | No (user needs to be on this page) |
| `/api/auth/callback` | Transitioning | Server-side callback handler | N/A (API route) |
| `/api/auth/signout` | Authenticated | Server-side sign out | N/A (API route) |

**Middleware behavior for auth flow routes**: Session refresh attempted. If user is already fully authenticated with completed onboarding and visits `/login` or `/signup`, redirect to their role dashboard. Exception: `/reset-password` is allowed even when authenticated (user arrived via recovery link).

## 4. Protected Routes (Session + Onboarding Required)

These routes require a valid session AND completed onboarding. Any authenticated user with completed onboarding can access them regardless of role.

| Route | Purpose | Fallback If No Auth | Fallback If No Onboarding |
|-------|---------|---------------------|---------------------------|
| `/dashboard` | Main dashboard | /login | /onboarding |
| `/profile` | User profile page | /login | /onboarding |
| `/profile/edit` | Edit profile | /login | /onboarding |
| `/settings` | User settings | /login | /onboarding |
| `/notifications` | Notification center | /login | /onboarding |
| `/onboarding` | Onboarding wizard | /login | N/A (this IS onboarding) |

**Special case: `/onboarding`**
- Requires authentication but NOT completed onboarding
- If onboarding is already completed, redirects to dashboard
- This is the only protected route that is accessible with incomplete onboarding

**Special case: `/dashboard`**
- All roles can access /dashboard
- The dashboard page itself renders role-specific content based on the user's role
- No RBAC check needed; the page queries role-appropriate data using RLS

## 7. Route Protection Mechanisms

Three mechanisms work together. They are layered, not alternatives.

### 7a. Middleware (`src/middleware.ts`)

The middleware runs on EVERY request (except static assets). It implements 7 layers in order:

```
Request
  |
  v
1. Session Refresh    -- Attempt to refresh the auth session from cookie
  |
  v
2. Security Headers   -- CSP, X-Frame-Options, etc.
  |
  v
3. Bot Blocking       -- Block known bad bots
  |
  v
4. Super Admin Guard  -- Extra protection for /super-admin/* routes
  |
  v
5. Rate Limiting      -- Upstash Redis (with in-memory fallback)
  |
  v
6. API Auth           -- For /api/* routes: validate session, check RBAC
  |
  v
7. Protected Pages    -- For page routes: check session, check onboarding
  |
  v
Response (or redirect)
```

**What middleware checks**:
- Is the route public? If yes, skip auth checks (layers 6-7).
- Does the user have a valid session? If no, redirect to /login (for pages) or return 401 (for APIs).
- Is the route an admin route? If yes, check for admin/super_admin role.
- Is the user's onboarding complete? If no and route requires it, redirect to /onboarding.

**What middleware does NOT check**:
- Fine-grained RBAC permissions (that is the API route's job via `authorizeRequest()`)
- Role-specific page access (handled by page components checking the user's role)

### 7b. API Route Authorization (`authorizeRequest()`)

Every API route that requires authentication calls `authorizeRequest()` as its first action.

```typescript
// Pattern used in every protected API route
export async function GET(request: Request) {
  const auth = await authorizeRequest(request, 'required.permission.code');
  if (!auth.authorized) {
    return Response.json({ error: 'Forbidden' }, { status: 403 });
  }
  // ... business logic
}
```

`authorizeRequest()` performs:
1. Extract session from request (via cookie or Authorization header)
2. Validate session with Supabase Auth
3. Look up user's role via `user_roles` table
4. Check if the role has the required permission via `role_permissions` table
5. Return `{ authorized: boolean, user, profile, role }`

### 7c. Client-Side Route Guards (UI Convenience Only)

Page components may check the user's role to render role-appropriate content or redirect. This is NOT a security boundary.

```typescript
// Example: page checks role for UI purposes
const { role } = useIdentityState();
if (role !== 'student') {
  redirect('/dashboard'); // UI redirect, not security
}
```

**This is defense-in-depth, not primary security.** If a parent somehow navigates to `/quiz`, the page might redirect them, but even if it does not, the quiz API endpoints will reject their requests via `authorizeRequest()` and RLS will prevent data access.

## 8. Redirect Rules

### Unauthenticated User Redirects

| User Visits | Redirected To | Reason |
|-------------|---------------|--------|
| Any protected route | `/login?redirect={original_path}` | Must authenticate first |
| Any role-restricted route | `/login?redirect={original_path}` | Must authenticate first |
| Any admin route | `/login` (no redirect param for admin) | Security: do not reveal admin URLs |
| `/login` (already on login) | No redirect | Already where they need to be |

The `redirect` query parameter is preserved so that after successful sign-in, the user is sent to their originally requested page (if they have access).

### Authenticated But Onboarding Incomplete

| User Visits | Redirected To | Reason |
|-------------|---------------|--------|
| Any protected route (except /onboarding) | `/onboarding` | Must complete setup first |
| Any role-restricted route | `/onboarding` | Must complete setup first |
| `/onboarding` | No redirect | Already where they need to be |
| `/profile` | `/onboarding` | Profile editing requires completed profile |
| Public routes | No redirect | Public routes are always accessible |

### Authenticated With Completed Onboarding

| User Visits | Redirected To | Reason |
|-------------|---------------|--------|
| `/login` | Role dashboard | Already authenticated |
| `/signup` | Role dashboard | Already authenticated |
| `/onboarding` | Role dashboard | Already onboarded |
| Wrong role's routes (e.g., student visits /parent/*) | `/dashboard` | Not their portal |
| Route they have permission for | No redirect | Allowed |

### Role Dashboard Destinations

This mapping MUST be defined in exactly one file: `src/lib/role-destinations.ts`.

| Role | Dashboard URL | Notes |
|------|--------------|-------|
| `student` | `/dashboard` | Main student dashboard |
| `parent` | `/parent/dashboard` | Parent portal |
| `teacher` | `/teacher/dashboard` | Teacher portal |
| `tutor` | `/tutor/dashboard` | Tutor portal |
| `admin` | `/super-admin` | Admin panel |
| `super_admin` | `/super-admin` | Admin panel |

### Post-Login Redirect Logic

After successful sign-in (Flow B in auth-contract.md):

```
1. Check for `redirect` query parameter
2. If redirect exists AND user has access to that route:
   -> Go to redirect URL
3. If redirect exists but user does NOT have access:
   -> Go to role dashboard (ignore redirect)
4. If no redirect:
   -> Go to role dashboard
```

### Post-Signup Redirect Logic

After successful sign-up and bootstrap (Flow A in auth-contract.md):

```
1. Always go to /onboarding (new user must complete setup)
2. No redirect parameter is honored for new signups
```

## 9. API Route Access Matrix

API routes follow the same classification but return HTTP status codes instead of redirects.

| Scenario | HTTP Response |
|----------|--------------|
| No session, accessing protected API | `401 Unauthorized` with `{ error: "Authentication required" }` |
| Valid session, insufficient permission | `403 Forbidden` with `{ error: "Insufficient permissions" }` |
| Valid session, onboarding incomplete, accessing feature API | `403 Forbidden` with `{ error: "Onboarding incomplete" }` |
| Valid session, correct permissions | `200` (or appropriate success code) |
| Rate limited | `429 Too Many Requests` with `{ error: "Rate limit exceeded", retryAfter: N }` |

## 10. Middleware Configuration

The middleware matcher determines which routes are processed by middleware.

```typescript
export const config = {
  matcher: [
    // Match all routes except static files and Next.js internals
    '/((?!_next/static|_next/image|favicon.ico|.*\\.(?:svg|png|jpg|jpeg|gif|webp|ico|css|js|woff|woff2|ttf|eot)$).*)',
  ],
};
```

### Routes excluded from middleware (static assets):
- `/_next/static/*` -- Next.js built assets
- `/_next/image/*` -- Next.js image optimization
- `/favicon.ico`
- Any file with a static file extension (`.svg`, `.png`, `.jpg`, `.css`, `.js`, `.woff2`, etc.)

### Routes with special middleware behavior:
- `/api/v1/health` -- No auth check, no rate limiting (health checks must always work)
- `/api/auth/callback` -- Auth state is transitioning; do not require valid session
- `/api/payments/webhook` -- No session auth (uses Razorpay signature verification instead)

## 11. Invariants

1. **No unclassified routes**: Every route MUST appear in this contract or be treated as Protected + Onboarding Required by default.
2. **Middleware is the first line**: All routes pass through middleware. No route bypasses it.
3. **API authorization is the second line**: Every API route calls `authorizeRequest()`. No API route skips this.
4. **RLS is the third line**: Even if middleware and API auth fail, RLS prevents unauthorized data access.
5. **Role destinations defined once**: The role-to-dashboard mapping exists in exactly one file (`src/lib/role-destinations.ts`). All redirect logic imports from there.
6. **No client-only security**: Client-side route guards are UI convenience. They do not replace server-side checks.
7. **Admin routes are invisible**: Admin route paths are not included in redirect parameters, sitemaps, or client-side route lists.
8. **Webhook routes use signature verification**: Payment webhook routes do not use session auth. They verify the Razorpay signature before processing any event (P11).
